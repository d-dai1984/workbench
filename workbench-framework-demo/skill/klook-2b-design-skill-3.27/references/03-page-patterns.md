# 03 · 标准页面模式

每个页面必须归属一种标准模式，不允许在没有模式定义的情况下直接堆砌组件。

---

## 工作台（Dashboard）

**结构**：顶部统计卡片区 + 中部图表/列表模块 + 快捷操作入口

```
Page Header（页面标题 + 欢迎语）
├── 统计卡片组（Statistic × n，4 列栅格）
├── 主内容区（图表 / 数据列表 / 任务模块）
└── 快捷入口（可选，最多 6 个）
```

规则：
- 统计卡片不超过 4 个一行
- 工作台不做复杂筛选，数据聚合展示为主
- 图表使用 Ant Design Charts（`@ant-design/charts`），文档：https://ant-design-charts.antgroup.com
- 快捷操作入口最多 6 个，位于统计卡片下方，用于跳转高频任务

---

## 列表页（List Page）

列表页分两种形态，根据数据结构选择：

### 形态一：标准列表（antd Table）

**适用**：每行数据结构扁平、无嵌套子项的场景。

**结构**：

```
Page Header
├── 页面标题
├── 副标题（可选，说明文字 + Learn more 链接，可配置）
└── 操作按钮组（右对齐）

筛选区
├── 搜索框（支持多字段关键词搜索）
└── Advanced Filters 按钮（点击展开更多筛选条件，收起式）

数据统计行（显示当前结果数量，如 "32 items"）

Table（主体）
└── Pagination（右对齐）
```

### 形态二：卡片列表（Card List）

**适用**：每行数据内部包含子项（卡片内有卡片）的场景，如 Combo 产品列表。

**结构**：

```
Page Header
├── 页面标题
├── 副标题（可选）
└── 操作按钮组（右对齐）

筛选区
├── 搜索框
└── Advanced Filters 按钮（收起式）

数据统计行

列表 Header（列名栏，固定）
Card × n（每张 Card 可展开子内容）
Pagination（右对齐）
```

### 通用规则（两种形态均适用）

- 副标题为可选项，Learn more 链接可配置，不强制显示
- 筛选区使用「搜索框 + Advanced Filters 收起式」，不默认平铺所有筛选字段
- 数据统计行显示当前筛选结果总数
- 操作列固定最右（`fixed: 'right'`）
- 行内操作全部平铺，不折叠进「更多」
- 支持批量操作时，批量操作栏出现在列表上方

## 详情页（Detail Page）

**结构**：面包屑 + 页面标题 + 状态栏（可选）+ 信息分区 + 操作区

```
Page Header
├── 面包屑（三级及以上显示）
├── 页面标题 + 状态 Tag
├── 操作按钮组（右对齐）
├── 基本信息卡片（Descriptions）
├── 关联信息模块（Table / Card）
└── 操作历史（Timeline，可选）
```

规则：
- 信息分区使用 Card 包裹，模块间有间距
- 不在详情页做复杂编辑，编辑走 Drawer 或跳转编辑页
- 操作按钮不超过 3 个主操作，超出折进 Dropdown

### 子模式：审核详情页

**适用场景**：Admin 内对商户、活动等内容进行审核决策的页面。

**结构**：

```
Page Header
├── 面包屑
├── 页面标题 + 当前状态 Tag（待审核 / 已通过 / 已拒绝）
├── 审核操作区（通过 + 拒绝，右对齐，仅「待审核」状态下可操作）
├── 基本信息卡片（Descriptions，只读）
├── 关联信息模块（可选）
└── 审核历史（Timeline，记录每次审核操作和备注）
```

**审核操作规则**：
- 「通过」使用 `type="primary"` Button，「拒绝」使用 `danger` Button
- 点击「拒绝」触发 Modal，Modal 内包含必填的「拒绝原因」输入框
- 拒绝原因字段：文本域（`TextArea`），最多 200 字，必填
- 确认拒绝后：关闭 Modal → 页面状态 Tag 更新 → 操作区按钮变为禁用
- 审核完成后操作区按钮置灰，不可重复提交
- 审核历史 Timeline 实时追加本次操作记录（操作人 + 时间 + 结论 + 备注）

**拒绝 Modal 结构**：

```
Modal
├── 标题：「拒绝审核」
├── 拒绝原因（TextArea，必填，最多 200 字）
└── 操作：取消 ／ 确认拒绝（danger）
```

## 表单页（Form Page）

表单页分两种形态，选择依据见下方规则。

### 形态一：单页表单

**适用**：字段相对集中、填写逻辑线性、用户可一次完成的场景。

**结构**：

```
Page Header
├── 面包屑
├── 页面标题
├── 表单区域（单列 / 两列 / Card 分组）
└── 底部操作区（取消 + 提交，右对齐，固定不随内容滚动）
```

字段量分级：
- ≤ 6 个字段 → 单列布局
- 7–12 个字段 → 两列或按主题分组
- > 12 个字段 → Card 分组，每组一个业务主题

### 形态二：多步骤向导（Steps + 分步表单）

**适用**：满足以下任一条件时使用 Steps 向导，否则默认使用单页表单：

| 条件 | 说明 |
|------|------|
| 字段总量 > 20 个 | 单页过长，认知负担重 |
| 不同步骤间存在强依赖 | 后一步的选项由前一步决定（如产品类型决定后续字段集） |
| 存在不可逆的中间提交节点 | 某一步完成后需要后端处理才能继续 |
| 完成率和流失率需要分段追踪 | 业务上需要知道用户在哪一步放弃 |

**结构**：

```
Page Header
├── 面包屑
├── 页面标题
├── Steps 进度条（横向，固定在表单上方）
├── 当前步骤表单内容（Card 包裹）
└── 步骤操作区（上一步 + 下一步 / 提交，右对齐，固定不随内容滚动）
```

规则：
- 最后一步必须有确认/预览环节，展示关键填写内容供用户核对
- 步骤数量 3–5 步为宜，不超过 6 步
- 每步的保存状态需要持久化，返回上一步不丢失已填内容
- 步骤之间的跳转只允许向后（已完成的步骤可点击回看，不可直接跳到未完成步骤）

### 通用规则（两种形态均适用）

- 底部操作区固定，不随内容滚动
- 表单校验走 Form inline 错误，不用全局 `message`
- 提交失败不关闭页面，保留已填内容并展示错误信息

## 配置页（Config Page）

**结构**：左侧导航（可选）+ 配置分区 + 保存操作

```
├── 左侧分类导航（Anchor / Menu，可选）
├── 配置区域（Card 分组）
└── 保存操作（固定在底部或 Card 内）
```

规则：
- 配置项按业务归类，不要一页平铺所有配置
- 危险操作（重置、清除）使用 `danger` 样式并加确认
- 保存成功/失败必须有明确反馈

## 选型决策

遇到新需求时，按以下顺序判断页面类型：

1. 主要目的是**展示列表数据并操作** → 列表页，再判断形态：
   - 数据结构扁平、无嵌套子项 → 标准列表（antd Table）
   - 数据内含子项（卡片内有卡片） → 卡片列表（Card List）
2. 主要目的是**查看单条数据详情**，且无审核决策 → 详情页
3. 主要目的是**查看详情并做审核决策**（通过 / 拒绝） → 详情页 · 审核子模式
4. 主要目的是**填写 / 编辑数据** → 表单页，再按以下条件选形态：
   - 满足 Steps 向导任一条件 → 多步骤向导
   - 不满足 → 单页表单
5. 主要目的是**系统 / 业务配置** → 配置页
6. 主要目的是**汇总展示多维数据** → 工作台
7. 以上均不符合 → 标记为 scope 外，评估后决定
