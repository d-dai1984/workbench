# 01 · 范围边界与准入规则

本 Skill 只处理已纳入范围的能力。每次任务开始前，先用本文件判断需求是否在范围内。

---

## 产品线说明

本 Skill 覆盖以下三条产品线，它们共享同一套设计系统，但服务对象和任务重心不同：

| 产品线 | 使用者 | 核心任务重心 | 当前状态 |
|-------|--------|------------|---------|
| **Admin** | Klook 内部运营 / 管理人员 | 商户管理、活动管理、订单管理、财务对账 | 主力在用 |
| **Merchant** | 商户（外部合作伙伴） | 产品上架（最高频）、订单处理、促销活动、结算查看 | 主力在用 |
| **Workbench** | Klook 内部员工 | 内部工作流，逐步替代 Admin | 2026 年启动，初期探索阶段 |

> Workbench 与 Admin 长期会合并职能，设计决策需考虑两者的兼容性。

## 已纳入范围的页面类型

### Admin 典型页面

| 业务域 | 主要页面形态 | 典型操作 |
|-------|------------|---------|
| 商户管理 | 列表页 · 详情页 · 表单页 · 配置页 | 审核、查找、查看、新建/编辑商户、设置权限等级 |
| 活动管理 | 表单页 · 列表页 · 详情页 · 配置页 | 创建编辑活动、审核活动、配置活动规则 |
| 订单管理 | 列表页 · 详情页 | 查看订单、搜索筛选 |
| 财务对账 | 列表页 | 查账、筛选、导出 |

> `[待补充]` Admin 还有哪些业务域（如用户管理、内容管理、系统配置等），待业务熟悉后补充。
> `[待补充]` 活动管理是否有独立数据报表页，待确认后补充页面形态。
> `[待补充]` 订单管理是否包含修改订单、退款处理等操作，待确认后补充典型操作。

### Merchant 典型页面

| 业务域 | 主要页面形态 | 典型操作 |
|-------|------------|---------|
| 产品上架 | 多步骤向导（Steps + 分步表单）· 单页长表单 | 填写产品信息、设置 SKU、提交审核 |
| 订单处理 | 列表页 · 详情页 | 查看订单、处理发货/取消 |
| 促销活动 | 列表页 · 表单页 | 创建促销、查看活动状态 |
| 结算收入 | 列表页 | 查看账单、筛选周期、导出 |

> `[待补充]` Merchant 还有哪些业务域，待业务熟悉后补充。

### Workbench 典型页面

当前处于初期探索阶段，已纳入范围的页面以工作台和内部任务流为主：

| 业务域 | 主要页面形态 | 说明 |
|-------|------------|------|
| 工作台 | 工作台（Dashboard） | 数据概览、任务入口、统计卡片 |
| 内部任务流 | 列表页 · 详情页 · 表单页 | 对应 Admin 职能的逐步迁移 |

> Workbench 新页面在未完成设计评估前，标记为「探索中」，不直接沉淀为正式规范。

## 已纳入范围的页面模式

以下五种页面模式在三条产品线内均适用：

| 页面模式 | 说明 |

## 超出范围的内容

以下需求**不在当前范围内**，需标记后评估：

| 类型 | 说明 |
|------|------|
| 官网 / 营销页 / 活动页 | C 端视觉风格，与 2B 系统不共享 |
| 面向 C 端的交互模式 | 不适用于 2B 操作密度和任务流 |
| 数据大屏 | 独立的全屏可视化场景，未纳入当前 Skill |
| 全新页面模式 | 无法映射到五种标准模式的需求 |
| 未经评估的新业务模块 | Workbench 新探索场景也在此列 |

**以下内容在范围内，但有独立规范：**

| 类型 | 说明 |
|------|------|
| 响应式页面 | 三条产品线需支持响应式，断点规则待补充至 `03-page-patterns.md` |
| 图表 / 数据可视化 | 使用 Ant Design Charts（`@ant-design/charts`）实现，文档：https://ant-design-charts.antgroup.com；图表规范待独立补充 |

## 准入判断流程

```
收到需求
  ↓
是否属于 Admin / Merchant / Workbench 三条产品线之一？
  ├── 否 → scope 外，不处理
  └── 是 ↓
      是否能归属五种标准页面模式之一？
        ├── 能 → 继续读取 03-page-patterns.md
        └── 不确定 → 尝试映射，仍无法归属则标记为「scope 外需求」
```

**标记原则**：
- 在范围内 → 按现有页面模式、模块和组件规则处理
- 部分在范围内 → 可处理部分，其余标记为 scope 外
- 完全不在范围内 → 明确说明原因，不输出长期规范
